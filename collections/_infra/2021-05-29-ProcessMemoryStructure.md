프로세스의 메모리 구조는 GC를 이해하기 전에 반드시 알아둬야 할 개념이다. 아래 유투브는 C 언어 기반으로 프로세스 메모리 구조를 설명하고 있는데,
설명이 아주 깔끔하게 잘 되어 있다. 여담이지만 불편함을 먼저 겪고 편함을 겪어야 원리를 아는데 더 좋은 것 같다.

처음부터 Java를 접했던지라 메모리 자체에 대해서도 인지가 약했고, GC가 어떤일을 하는지 등에 대해서도 둔감했는데,
이 영상을 보니 애초에 처음부터 C로 프로그래밍을 접했으면 후에 Java를 하더라도 GC에 대해서 감사해하며 프로그래밍을 했을 것 같다.

보통 다른 블로그들을 보면 영상과는 다르게 code -> data -> .. 순으로 정리가 된게 많은데 이 영상에서는 반대의 순서로 되어 있었다.
익숙하게 알던 순서와는 반대로 나와 있으니 아무래도 처음에는 좀 헷갈렸는데 보다보니 이게 더 기억해두기 편한 것 같다.

위 유투브에 나온 순으로 정리를 해보자면 아래와 같다.

<br>

(1) **<blue>code</blue>**<br>
instruction 들이 보관된다. "need to be executed". 여기 보관된 instruction 의 실행에 의해서 개별 실행이 발생한다.

<br>

(2) **<blue>static / global</blue>**<br>
어떤 레퍼런스에서는 data라고 표현되어 있고 조금 세분화해서 값이 할당이 된 것과 그렇지 않은 것까지 구분해두었던데, 나는 영상 그대로를 따라 정리하려고 한다.
이 영역에는 어플리케이션의 전체의 라이프타임에 언제든지 접근이 가능한 변수들이 저장된다.

<br>

(3) **<blue>stack</blue>**<br>
function calls와 지역 변수들이 저장되는 곳이다. 여기서 function call이 쌓인다는 것을 아는 것이 중요하다. 개별 함수들이 실행이 되면
이 영역에 쌓이게 되는데 실행된 함수 내에서 또다른 함수가 실행이 되면 해당 함수 위에 stack 으로 뒤에 실행된 함수가 추가로 쌓이게 된다.
실행되는 순서대로 stack으로 쌓이게 되고 종료되는 순으로 pop이 발생한다.

그래서 재귀로 계속 함수를 호출하면서 종료조건을 주지 않으면 stack으로 계속 쌓이기만하고 결국 'stack over flow' 가 발생하게 된다.
영상에서 2:15부터 그림과 함께 자세하게 순서대로 보여주고 있는데 좀 재밌다.

function calls 외에 또 중요한 사항은 stack에는 지역 변수가 저장된다는 것이다. 가령 String testText = "test" 라고 선언해주면
testText의 주소값이 stack에 저장되고 그 주소값은 heap을 가리키고 있게 된다.

<br>

(4) **<blue>heap</blue>**<br>
동적으로 할당되는 영역이다. 객체를 선언하고 이 타입에 맞게 객체를 할당해주면 실제로 할당된 객체가 들어가 있게 되는 공간이다.
나중에 정리할 GC가 활동하는 공간이기도 하다.
