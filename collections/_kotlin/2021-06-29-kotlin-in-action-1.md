## 1.1 코틀린 맛보기
```kotlin
fun main() {
    val persons = listOf(Person("영희", 29), Person("철수", 29))
    val oldest = persons.maxByOrNull { it.age ?: 0 }
    println("나이가 가장 많은 사람 : $oldest")
}

data class Person(val name: String, val age: Int? = null)
```

간단한 예제로 미리 코틀린을 접할 수 있었다. 짚고 넘어가야할 몇몇의 포인트들이 있었는데, 어차피 나중에 다룰 것이기에
이번에는 넘어간다.

<br>

## 1.2 코틀린의 주요 특성
### 1.2.1 대상 플랫폼: 서버, 안드로이드 등 자바가 실행되는 모든 곳
코틀린의 탄생배경 자체가 자바의 장점은 그대로 살리되 단점을 개선하기 위함이므로
코틀린을 적용할 수 잇는 플랫폼은 자바가 작동하고 있는 모든 플랫폼이라 할 수 있겠다.

뿐만 아니라 자바스크립트로도 컴파일이 가능하다. 'Kotlin for JavaScript' 라는 제목으로
[공식문서](https://kotlinlang.org/docs/js-overview.html) 에서도 다루고 있다.

이건 한번도 직접 써보지 못했고, 쓰는 것을 본적도 없어서 효율에 대한 판단이 잘 서질 않는다.
만약 호환이 100% 잘된다면, 허술한 js로 프론트 로직 짜는 것 보다 코틀린으로 서버사이드랑 프론트랑 같이
코딩하는 것도 나쁘지 않겠다는 생각이 들었다.

<br>

### 1.2.2 정적 타입 지정 언어
자바와 같이 코틀린은 **<red>statically typed</red>** 언어로 컴파일 시점에 타입에 대한 검증이 이뤄진다.
반대로 **<red>dynamically typed</red>** 언어는 런타임 시점에 타입에 대한 검증이 이뤄진다.
사실 검증이 이뤄진다는 표현 보다는 안맞으면 오류가 난다는 표현이 더 맞는 듯 하다.

그래서 statically typed 언어는 dynamically typed 언어에 비해서 엄격함을 더 지켜야 하기 때문에
개발 유연성이 다소 떨어진다고 할 수는 있겠으나 그만큼 자유도를 비용으로 지불하고 안정성을 보장하게 된다.

대신 코틀린은 [타입 추론](https://en.wikipedia.org/wiki/Type_inference) 을 제공해주기 때문에
statically typed 언어임에도 조금은 유연하게 코드를 작성할 수 있다.
컴파일러가 문맥을 고려해서 타입을 결정해줄 수 있기 때문에 명시적인 타입을 생략할 수 있기 때문이다.

타입에 관해서 자바에서 지원해주지 않는 새로운 것들이 있는데 nullable type 과 function type 이다.
* 후에 다시 다루게 될거라서 자세히 짚진 않겠지만 nullable type 은 명시적으로 null 을 허용할지를 미리 결정함으로써
적어도 컴파일 시점에 NPE(NullPointerException)을 검사할 수 있게 한다.
* function type 은 말 그대로 함수가 곧 타입인 것인데 아래에서 자세히 다루고 있다.

<br>

### 1.2.3 함수형 프로그래밍과 객체지향 프로그래밍
자바도 함수형 프로그래밍이 가능은 하지만 자바 8 이전에는 함수형 프로그래밍을 지원할 수 있는 기능이 거의 없었다고 한다.
반면에 코틀린은 처음부터 함수형 프로그래밍을 풍부하게 지원했다고 한다. 그래서 상황에 맞게 객체 지향과 함수형 접근 방법을 조합하여
문제에 가장 적합한 도구를 사용하길 권장한다.

## 1.4 코틀린의 철학
### 1.4.1 실용성
탄생 배경 자체가 큰 규모의 프로젝트에서 자바가 갖는 한계의 개선이었기 때문에, 실용성을 추구하며 나아가서
IDE 회사가 만들었기 때문에 인텔리J의 서포트가 강력하므로 실용적이라 할 수 있다.

<br>

### 1.4.2 간결성
여러가지 표준 라이브러리를 제공함으로써 쓸데 없이 만들거나 작성하지 않고도 원하는 기능이 구현 가능하고,
자바에서 자주 사용되었던 준비 코드들을 코틀린에서는 묵시적으로 제공하도록 함으로써 필요한 코드만 넣고 의미 없는 코드는
줄이는 방향으로 간결성을 확보했다.

책에서는 컬렉션에서 제공해주는 자원들을 제공해주는 여러가지 표준 라이브러리의 예시로 들고 있었는데,
나중에 컬렉션 파트에서 깊게 파봐야겠다는 생각이 들었다.

<br>

### 1.4.3 안전성
안전성과 생산성이 트레이드오프 관계인 것에 대한 설명이 먼저 이뤄졌는데, 내용이 상당히 상식적이며 괜찮았다.
안전하다는 것은 곧 최대한 원천적으로 개발자가 범할 수 있는 오류를 사전에 알려주고 발견한다는 의미인데, 다르게 설명하자면
실행하다가 오류가 나지 않도록 컴파일 타임에 이를 알려준다는 것이다.

**<red>하지만 컴파일 타임에 오류가 나지 않도록 하려면 그만큼 컴파일러가 판단해줘야 할 포인트들이 많고
이 판단을 위해서 개발자 역시 코드를 더 쓸 수 밖에 없기에 안전성과 생산성은 트레이드오프 관계라는 논리인데
합리적인 이야기였다.</red>**

이 부분에서 코틀린이 강조하는 것은 코틀린은 [타입 추론](https://en.wikipedia.org/wiki/Type_inference) 을 제공해주기 때문에
안전성을 챙기기위한 비용을 자바보다는 조금이라도 덜 지불했다는 것이었다.

그리고 **<red>nullable type</red>** 의 도입으로 NPE를 최대한 예방한다는 것 역시 코틀린이 안전성을 제공해준다는 근거 중 하나다.

ClassCastException 을 예방해주는 것 역시 코틀린이 안전성을 제공해준다는 근거 중 하나인데, **<red>타입 검사와 캐스트가 한 연산자에 의해 이뤄져서
별도의 캐스트를 해줄 필요가 없이 타입 검사만 하면 안전하게 타입에 맞는 코딩을 할 수 있다.</red>**

```kotlin
if (value is String) // 타입을 검사
    println(value.toUpeerCase()) // 타입 검사와 함께 캐스트가 되었기에 .toUpeerCase()를 사용할 수 있음
```

<br>

### 1.4.4 상호운용성
코틀린의 근간 자체가 자바의 개선임과 동시에 자바와의 상호운용성에 포커스가 있기 때문에 이는 코틀린의 가장 중요한 핵심적인 가치라 할 수 있다.
**<red>일례로 코틀린은 자체 컬렉션 라이브러리를 제공하지 않고, 자바 표준 라이브러리 클래스를 의존하기 때문에 코틀린에서 자바API를 호출 할 때
뭔가 추가적으로 자바 API를 감싸게 된다던가 하는 추가적인 비용이 들지 않는다.</red>**

비용적인 부분 뿐만 아니라 상호운용성이 완벽하게 보장되다보니 하나의 프로젝트 내에서 코틀린과 자바를 편하게 섞어서 쓰는 것이 가능하고
그만큼 코틀린의 점진적인 도입, 경우에 따른 취사선택등 유연한 개발이 가능하다.

## 1.5 코틀린 도구 사용
코틀린의 컴파일이 어떻게 이뤄지는지 알아본다.

<br>

### 1.5.1 코틀린 코드 컴파일
{% include image.html src="/assets/images/kotlin/kotlin-build-process.png" %}
코틀린 컴파일러가 *.kt 파일을 *.class 파일로 컴파일해주고 그 이후 단계는 자바와 동일하다.
그래서 kotlinc 를 이용해서 컴파일한 class파일을 java 명령어로 실행이 가능하다.

<br>

### 1.5.2 인텔리J 아이디어와 안드로이드 스튜디오의 코틀린 플러그인
코틀린 컴파일러의 개발과 이에 맞는 인텔리J 개발이 같이 이뤄졌기 때문에 호환이 좋다고 책 전반에서 여러번 강조하고 있다.
그런 맥락의 내용이었다.

<br>

### 1.5.3 대화형 셸
REPL(read-eval-print-loop) 를 통해서 바로 코틀린 사용이 가능하다. 아무런 인자를 주지 않고 kotlinc 명령어를 통해서
REPL 환경으로 접속할 수 있고 거기서 kotlin 코딩이 가능했다. 해보니까 실제로 되긴 하는데(당연히 되겠지만) 이걸 쓸일이 있을까 싶었다.
그런데 한번씩 콘솔에서 뭔가 해봐야하는 상황이 생기긴 해서 실무 하다가 사용해볼 것 같기도 했다.

<br>

### 1.5.4 이클립스 플러그인
이클립스에서도 사용이 가능하다는 이야기였다.

<br>

### 1.5.5 온라인 놀이터
try.kotl.in 에서 바로 코틀린 사용이 가능하다. 브라우저 상에서 IDE가 켜지는 거라서 REPL 보다 더 시각적이어서 좋다.

<br>

### 1.5.6 자바-코틀린 변환기
자바-코틀린간 코드를 자동으로 변환해주는 것인데 자바에 능숙하지만 코틀린이 헷갈리거나 그 반대의 경우 코드를 알고 싶을 때 써보라는 이야기였다.
